# Docs Tutorial: Submit PR

在你已经确认本地 `master` 是最新的，并且已经完成文档修改之后，就可以开始开分支、提交修改并发起 PR 了。

## 1. 提交流程概览

这一篇主要处理下面这段流程：

```text
创建新分支 -> 修改后保存 -> 提交修改 -> 推送到远端 -> 创建 PR -> 等待审核 -> 合并后清理本地分支
```

---

## 2. 创建新的分支

### 说明

- 这一篇默认你当前的 `master` 已经提前同步到最新。
- 新分支通常从最新的 master 拉出来。

 > [!NOTE]  
 > 以后当这个 PR 合并完成后，要记得切回 master 分支。

### 操作步骤

1. 点击左下角当前显示的分支名称。

![点击左下角的分支名称](/Docs%20Tutorial/Assets/submit-branch-button.png)

图 8：创建新分支时，先点击左下角显示的分支名称。
2. 选择 `创建新分支`。
3. 输入分支名，例如：

![选择创建新分支并输入分支名](/Docs%20Tutorial/Assets/submit-create-branch-dialog.png)

图 9：选择“创建新分支”后，输入分支名。

```text
feature/login-page
fix/user-export
docs/update-readme
```

4. 创建后，VS Code 会自动切换到新分支。

### 分支名字建议

- `feature/功能名`
- `fix/问题名`
- `docs/文档名`
- `refactor/重构名`

---

## 3. 保存修改并提交

### 操作步骤

1. 修改文档内容后，先按 `Ctrl + S` 保存文件。
2. 点击左侧 `源代码管理`。
3. 确认变更文件显示在列表中。
4. 在顶部输入提交说明。
5. 点击 `提交`。

![源代码管理中保存后提交的示意图](/Docs%20Tutorial/Assets/submit-save-and-commit.png)

图 10：图中标出了第 2、4、5 步，对应“点击源代码管理”“输入提交说明”“点击提交”。

### 提交说明示例

```text
docs: 更新安装说明
docs: 修正文档截图说明
docs: 补充提交流程
```

### 说明

- 提交前先保存文件，否则最新修改内容可能不会被带上。
- 文档类修改的提交说明，建议以 `docs:` 开头。
- 如果 VS Code 提示你需要一并处理变更，按界面提示继续即可。

---

## 4. 把分支上传到远端

当你刚创建好一个新分支，并且已经提交过修改后，还需要把这个分支上传到远端仓库。只有上传成功后，后面才能发起 PR。

### 操作步骤

1. 先确认你已经完成了上一步“保存修改并提交”。
2. 点击左侧 `源代码管理`。
3. 观察界面中是否出现 `Publish Branch`、`发布分支` 或类似按钮。
4. 如果看到了这个按钮，直接点击它。

![Publish Branch 按钮位置](/Docs%20Tutorial/Assets/submit-publish-branch-button.png)

图 11：图中标出了 `Publish Branch` 按钮的位置。
5. 如果点击后弹出 GitHub 登录窗口，这是正常现象。
6. 直接点击 `Sign in with your browser`。
7. 按浏览器中的提示完成登录和授权。
8. 登录完成后，回到 VS Code，继续发布分支。
9. 如果界面没有自动继续，就再点一次 `Publish Branch` 或 `Push`。
10. 如果一开始就没有看到这个按钮，再点击右上角 `...`。
11. 在菜单中选择 `Push`。
12. 等待上传完成。

### 你可以这样理解

- `Publish Branch` 的意思就是“把这个新分支第一次上传到远端”。
- 对于新建分支，很多时候 VS Code 会优先显示 `Publish Branch`。
- 如果没有看到 `Publish Branch`，再去菜单里找 `Push` 即可。

### 说明

- 第一次上传新分支时，界面显示 `Publish Branch` 是正常的。
- 上传成功后，远端仓库里就会出现同名分支。
- 只有这一步完成后，后面才能继续创建 PR。

---

## 5. 在 GitHub 网页中发起 PR

分支上传完成后，建议直接到 GitHub 网页中发起 PR。这样更直观，也不需要额外安装 VS Code 插件。

### 操作步骤

1. 打开 GitHub 上 `RMS-Docs` 的项目主页。
2. 找到页面中的 `Pull requests` 入口。

![GitHub 中 RMS-Docs 项目主页和 PR 入口](/Docs%20Tutorial/Assets/pr-github-home-and-pr-entry.png)

图 12：GitHub 中 `RMS-Docs` 项目主页，以及 `Pull requests` 的位置。

3. 进入 PR 页面后，选择用你刚刚上传的新分支创建 PR。
4. 确认源分支是你刚刚提交的分支。
5. 确认目标分支是 `master`。

![在 PR 界面用新分支创建 PR](/Docs%20Tutorial/Assets/pr-create-from-branch.png)

图 13：在 PR 界面中，使用新分支创建 PR。

6. 根据本次更新内容，填写 PR 标题和说明。
7. 点击创建 PR 的按钮提交。

![填写 PR 标题说明并提交](/Docs%20Tutorial/Assets/pr-title-description-and-submit.png)

图 14：根据更新内容，填写 PR 标题和说明，并点击提交按钮。

### PR 描述建议

```text
## 变更内容
- 更新了哪些文档
- 补充了哪些说明

## 影响范围
- 涉及哪些页面或章节

## 自查情况
- 已检查格式
- 已检查链接或图片引用
```

### 说明

- 这里建议直接使用 GitHub 网页操作，不依赖 VS Code 插件。
- 创建前，确认目标分支是 `master`。
- PR 发起完成后，就进入等待审核和合并阶段。

---

## 6. 等待 PR 合并

PR 发起后，等待审核和合并即可。

### 说明

- 如果仍需修改，就继续在原来的分支上改、保存、提交并推送。
- 新的提交会自动进入同一个 PR，不需要重新创建新的 PR。

---

## 7. PR 合并后，先切回 master

### 这一步很重要

 > [!NOTE]  
 > PR 合并完成后，第一件事就是切回 `master` 分支。


### 操作步骤

1. 点击左下角分支名称。
2. 选择 `master`。
3. 确认当前左下角显示的分支已经变成 `master`。
4. 然后再执行 `Pull`，同步最新内容。
5. 如果不再需要旧分支，可删除本地分支。

### 删除本地分支的图形化操作

1. 点击左下角分支名称。
2. 选择分支管理。
3. 删除已合并的功能分支。

---

## 8. 常见问题

### 1. 为什么不能直接在 master 上开发

因为多人协作时，直接在主分支开发风险较高，容易影响他人，也不方便审核。

### 2. 审核意见修改后需要重新开 PR 吗

不需要。在原分支继续提交并推送即可。

### 3. 创建 PR 时看不到我的分支

可能原因：

- 分支还没有推送到远端。
- 推送失败。
- 当前仓库页面未刷新。
